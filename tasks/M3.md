# M3 Dashboard Task List

Source: `implementation-plan.md`

This milestone renders the MVP dashboard from live activity data and shared calculation modules.

## Task 1: Dashboard Calculation Adapters

**Objective**

Create pure dashboard data adapters for all chart panels so chart components stay presentation-focused.

**Files Affected**

- `lib/calculations/efficiency.ts`
- `lib/calculations/weekly-mileage.ts`
- `lib/calculations/rolling-average.ts`
- `lib/calculations/dashboard.ts`
- `tests/unit/dashboard-calculations.test.ts`
- `tests/unit/weekly-mileage.test.ts`
- `tests/unit/efficiency.test.ts`

**Acceptance Criteria**

- Pace/HR chart data covers eligible runs in the last 90 days.
- Aerobic efficiency chart data covers eligible runs in the last 6 months.
- VO2 chart data skips null and outlier values.
- Weekly mileage includes the last 16 weeks and fills empty weeks with zero.
- Recent runs data returns the last 10 activities.
- Pure functions have no Supabase dependency.

**Tests Required**

- Unit tests for pace/HR chart shaping.
- Unit tests for aerobic efficiency trend shaping.
- Unit tests for VO2 null and outlier filtering.
- Unit tests for weekly mileage empty weeks.
- Unit tests for recent runs ordering and limiting.

## Task 2: Dashboard Shell

**Objective**

Build the dashboard composition that fetches activity data, handles shared states, and lays out all dashboard panels.

**Files Affected**

- `components/dashboard/dashboard.tsx`
- `app/page.tsx`
- `tests/e2e/upload-dashboard-chat.spec.ts`

**Acceptance Criteria**

- Dashboard renders above or near the primary app surface.
- Loading state is visible while activities load.
- Fetch error state includes a retry path.
- Empty database state points the user to upload.
- One insufficient-data panel does not block other panels from rendering.

**Tests Required**

- Component test for loading state.
- Component test for empty state.
- Component test for fetch error state.
- Playwright test verifying dashboard shell renders after data load.

## Task 3: Pace vs Heart Rate Chart

**Objective**

Render pace and heart-rate trends over time for eligible recent runs.

**Files Affected**

- `components/dashboard/pace-heart-rate-chart.tsx`
- `lib/calculations/dashboard.ts`
- `tests/unit/dashboard-calculations.test.ts`
- `tests/e2e/upload-dashboard-chat.spec.ts`

**Acceptance Criteria**

- Chart uses eligible running activities from the last 90 days.
- Pace displays in min/km.
- Heart rate displays in bpm.
- Pace axis treats lower pace as faster.
- Insufficient data shows a panel-level message.

**Tests Required**

- Unit tests for generated chart points.
- Component test for populated chart state.
- Component test for insufficient-data state.
- Playwright assertion that the panel is visible with seeded data.

## Task 4: Aerobic Efficiency Trend Chart

**Objective**

Render the core fitness trend chart using the shared aerobic efficiency definition.

**Files Affected**

- `components/dashboard/efficiency-trend-chart.tsx`
- `lib/calculations/efficiency.ts`
- `lib/calculations/dashboard.ts`
- `tests/unit/efficiency.test.ts`
- `tests/unit/dashboard-calculations.test.ts`
- `tests/e2e/upload-dashboard-chat.spec.ts`

**Acceptance Criteria**

- Chart uses eligible running activities from the last 6 months.
- Efficiency is calculated as `speed_m_per_sec / avg_hr`.
- Raw run points and 30-day rolling average data are available to render.
- Reference data for the 90-day-ago rolling average is available when enough data exists.
- Panel shows "Not enough data yet" when eligibility filters leave too few runs.

**Tests Required**

- Unit tests for eligibility filters.
- Unit tests for efficiency trend points.
- Unit tests for 30-day rolling average.
- Component test for populated efficiency chart state.
- Component test for insufficient-data state.

## Task 5: VO2 Max Trend Chart

**Objective**

Render VO2 max trend data while handling missing Garmin VO2 values safely.

**Files Affected**

- `components/dashboard/vo2-trend-chart.tsx`
- `lib/calculations/dashboard.ts`
- `tests/unit/dashboard-calculations.test.ts`
- `tests/e2e/upload-dashboard-chat.spec.ts`

**Acceptance Criteria**

- Chart uses all-time VO2 data.
- Null VO2 values are skipped.
- Values below 30 or above 80 are excluded as outliers.
- 7-run rolling average data is available when enough data exists.
- Fewer than 5 non-null values shows "Not enough data yet."

**Tests Required**

- Unit tests for null handling.
- Unit tests for outlier filtering.
- Unit tests for 7-run rolling average.
- Component test for insufficient VO2 data.
- Playwright assertion that VO2 panel renders with seeded data.
