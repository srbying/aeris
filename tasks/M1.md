# M1 Data Pipeline Task List

Source: `implementation-plan.md`

Prerequisite: Milestone 0 project foundation is complete, including Next.js, TypeScript strict mode, Tailwind, Vitest, Playwright, Supabase configuration, and baseline app structure.

## Task 1: Activity Domain Schema

**Objective**

Define the canonical activity types and validation schemas that every parser, route, repository, dashboard, and chat feature will share.

**Files Affected**

- `lib/activity/types.ts`
- `lib/activity/schema.ts`
- `lib/config/env.ts`
- `.env.example`
- `tests/unit/activity-schema.test.ts`
- `tests/unit/env-config.test.ts`

**Acceptance Criteria**

- Activity input, stored activity, and public API activity shapes are explicit TypeScript types.
- Zod validation covers required fields, nullable optional fields, and upload payload shape.
- `ACTIVITY_CONTEXT_MONTHS=12` is documented and validated as a configurable default.
- Invalid activity rows fail validation with useful reasons instead of silent coercion.

**Tests Required**

- Unit tests for valid activity input.
- Unit tests for missing required fields.
- Unit tests for nullable HR, max HR, calories, ascent, and VO2 max fields.
- Unit tests for environment defaults and invalid `ACTIVITY_CONTEXT_MONTHS` values.

## Task 2: Garmin CSV Parser

**Objective**

Parse Garmin Connect CSV rows into validated Aeris activity inputs while preserving raw source rows and clear skip reasons.

**Files Affected**

- `lib/activity/garmin-parser.ts`
- `lib/activity/schema.ts`
- `tests/unit/garmin-parser.test.ts`
- `tests/fixtures/garmin-activities.csv`

**Acceptance Criteria**

- Garmin fields map to Aeris fields for date, activity type, distance, duration, pace, HR, calories, ascent, VO2 max, and raw row data.
- `--` values become `null`, not `0`.
- Rows missing `activity_date`, `distance_km`, or `duration_seconds` are skipped with a reason.
- Parser keeps enough raw data to diagnose field-name or unit issues later.
- Unit assumptions for distance and pace are explicit in parser behavior and tests.

**Tests Required**

- Unit tests for valid Garmin rows.
- Unit tests for optional missing fields.
- Unit tests for `--` null conversion.
- Unit tests for invalid required fields and skipped-row reasons.
- Unit tests for duration and pace parsing.

## Task 3: Activities Table Migration

**Objective**

Create the Supabase `activities` table and unique deduplication constraint required by the upload pipeline.

**Files Affected**

- `supabase/migrations/0001_create_activities.sql`
- `supabase/seed.sql`
- `tests/integration/activity-repository.test.ts`

**Acceptance Criteria**

- Table contains every key column from the PRD.
- `id` defaults to a UUID.
- `created_at` defaults to the insert timestamp.
- Unique index on `(activity_date, activity_type, distance_km)` exists.
- No RLS policies are required for MVP.

**Tests Required**

- Integration test proving inserts succeed with valid rows.
- Integration test proving duplicate rows hit the unique constraint path.
- Migration smoke check against the local Supabase test database.

## Task 4: Activity Repository Insert Flow

**Objective**

Implement repository-level insert/upsert behavior so route handlers stay thin and duplicate accounting is centralized.

**Files Affected**

- `lib/activity/activity-repository.ts`
- `lib/supabase/server.ts`
- `lib/activity/types.ts`
- `tests/integration/activity-repository.test.ts`

**Acceptance Criteria**

- `insertActivities(rows)` returns `{ inserted, skipped, errors }`.
- Duplicate rows are counted as skipped, not surfaced as fatal errors.
- Parse/validation skips and database skips are distinguishable in the result.
- Supabase access is hidden behind the repository layer.

**Tests Required**

- Integration tests for all-new insert.
- Integration tests for duplicate re-upload.
- Integration tests for mixed valid, duplicate, and invalid rows.
- Integration tests for repository error handling.

## Task 5: Upload API Route

**Objective**

Expose `POST /api/upload` for CSV ingestion, validation, deduplication, and user-facing import summaries.

**Files Affected**

- `app/api/upload/route.ts`
- `lib/activity/garmin-parser.ts`
- `lib/activity/activity-repository.ts`
- `lib/activity/schema.ts`
- `tests/integration/upload-route.test.ts`

**Acceptance Criteria**

- Route accepts `multipart/form-data` with field name `file`.
- Files over 10MB are rejected before parsing.
- Unrecognized CSV shape returns a clear Garmin-export error.
- Successful uploads return `{ inserted, skipped, errors }`.
- Supabase failures return a clear retryable upload error.
- Route uses `export const dynamic = 'force-dynamic'`.

**Tests Required**

- Integration test for successful upload.
- Integration test for re-upload with duplicates.
- Integration test for unrecognized CSV format.
- Integration test for oversized file.
- Integration test for Supabase failure.

## Task 6: Upload UI

**Objective**

Build the upload panel that lets the user select or drag a Garmin CSV and see progress, success, and error feedback.

**Files Affected**

- `components/upload/upload-panel.tsx`
- `components/upload/file-dropzone.tsx`
- `app/page.tsx`
- `tests/e2e/upload-dashboard-chat.spec.ts`

**Acceptance Criteria**

- User can select a CSV through a file picker.
- User can drag and drop a CSV.
- Upload in progress is visible.
- Success state shows inserted and skipped counts.
- Error state clearly explains unrecognized CSV and network failure cases.
- Re-uploading the same file visibly reports skipped duplicates.

**Tests Required**

- Component test for file selection state.
- Component test for upload success summary.
- Component test for upload error state.
- Playwright flow verifying visible inserted/skipped feedback.
