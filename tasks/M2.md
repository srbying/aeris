# M2 Chat Task List

Source: `implementation-plan.md`

This milestone folds in the shared retrieval and calculation work needed for chat. Dashboard-specific rendering remains in `tasks/M3.md`.

## Task 1: Activity Retrieval API

**Objective**

Expose normalized persisted activities so chat context and dashboard consumers can depend on one public activity contract.

**Files Affected**

- `app/api/activities/route.ts`
- `lib/activity/activity-repository.ts`
- `lib/activity/serializers.ts`
- `lib/activity/types.ts`
- `tests/integration/activities-route.test.ts`

**Acceptance Criteria**

- `GET /api/activities` returns normalized activity JSON matching the technical plan contract.
- Activities are sorted consistently by activity date.
- API fields use public camelCase names.
- Route uses `export const dynamic = 'force-dynamic'`.
- Repository access remains outside route business logic.

**Tests Required**

- Integration test for response shape.
- Integration test for sorting.
- Integration test for nullable fields.
- Integration test for repository failure behavior.

## Task 2: Configurable Chat Context Window

**Objective**

Make chat context filtering depend on `ACTIVITY_CONTEXT_MONTHS=12` by default instead of a hardcoded 12-month decision.

**Files Affected**

- `lib/config/env.ts`
- `lib/activity/activity-repository.ts`
- `lib/llm/context.ts`
- `.env.example`
- `tests/unit/env-config.test.ts`
- `tests/unit/prompt-context.test.ts`

**Acceptance Criteria**

- `ACTIVITY_CONTEXT_MONTHS` defaults to `12`.
- Invalid values fail validation.
- Chat context builder passes the configured month value to repository retrieval.
- Only running activities inside the configured window enter the model context.

**Tests Required**

- Unit test for default context window.
- Unit test for overridden context window.
- Unit test for invalid env value.
- Unit test proving out-of-window activities are excluded.

## Task 3: Chat Metric Context

**Objective**

Precompute aerobic efficiency snapshots and compact activity context so the model receives consistent, bounded data.

**Files Affected**

- `lib/calculations/efficiency.ts`
- `lib/calculations/rolling-average.ts`
- `lib/activity/serializers.ts`
- `lib/llm/context.ts`
- `tests/unit/efficiency.test.ts`
- `tests/unit/prompt-context.test.ts`

**Acceptance Criteria**

- Aerobic efficiency uses `speed_m_per_sec / avg_hr`.
- Eligibility filters match the implementation plan.
- Context includes 30-day, 90-day, and 180-day efficiency snapshots.
- Recent activities serialize to compact keys such as `d`, `pace`, `hr`, `dist`, and `vo2`.
- Missing or sparse data produces `null` snapshots instead of fabricated values.

**Tests Required**

- Unit tests for efficiency formula.
- Unit tests for eligibility filtering.
- Unit tests for rolling snapshot calculations.
- Unit tests for compact activity serialization.
- Unit tests for sparse-data/null behavior.

## Task 4: LLM Provider Abstraction

**Objective**

Implement the provider interface and OpenAI provider while preserving a clean future swap point.

**Files Affected**

- `lib/llm/types.ts`
- `lib/llm/openai.ts`
- `lib/llm/ollama.ts`
- `lib/llm/index.ts`
- `.env.example`
- `tests/integration/chat-route.test.ts`

**Acceptance Criteria**

- Provider interface streams text deltas as an async iterable.
- OpenAI provider uses the Responses API.
- Provider is selected with required `LLM_PROVIDER`, e.g. `openai`.
- Model is selected with required `LLM_MODEL`, e.g. `gpt-5.5`.
- Missing `LLM_PROVIDER` or `LLM_MODEL` throws a configuration warning.
- Route code depends on the provider interface, not OpenAI-specific code.
- Ollama remains optional and isolated for local experimentation.

**Tests Required**

- Unit or integration test for provider factory selection.
- Integration test with mocked OpenAI streaming response.
- Integration test for missing `OPENAI_API_KEY`.
- Integration test proving route code can use a mocked provider.

## Task 5: Aeris Prompt Contract

**Objective**

Store the Aeris system prompt as versioned product logic with explicit constraints against hallucination and coaching.

**Files Affected**

- `lib/llm/prompts.ts`
- `lib/llm/context.ts`
- `tests/unit/prompt-context.test.ts`

**Acceptance Criteria**

- Prompt includes `PROMPT_VERSION`.
- Prompt instructs the model to answer only from supplied running data.
- Prompt requires relevant dates or time periods when making claims.
- Prompt requires explicit uncertainty when data is noisy or sparse.
- Prompt excludes coaching recommendations and training plans.

**Tests Required**

- Unit test for prompt version presence.
- Unit test for injected efficiency stats.
- Unit test for injected compact activity JSON.
- Unit test for no-coaching and no-training-plan constraints.
- Unit test for uncertainty instruction.

## Task 6: Chat API Route

**Objective**

Expose `POST /api/chat` as an Edge Runtime streaming route that validates input, builds context, calls the LLM router, and emits SSE events.

**Files Affected**

- `app/api/chat/route.ts`
- `lib/llm/context.ts`
- `lib/llm/index.ts`
- `lib/activity/schema.ts`
- `tests/integration/chat-route.test.ts`

**Acceptance Criteria**

- Route declares `export const runtime = 'edge'`.
- Request body validates `{ message, history }`.
- No uploaded activities returns the upload CTA error.
- Successful response streams `{ delta }` events.
- Final event is `{ done: true }`.
- Provider timeout and 5xx errors surface user-safe messages.

**Tests Required**

- Integration test for valid streaming response.
- Integration test for invalid request body.
- Integration test for no activities uploaded.
- Integration test for provider timeout.
- Integration test for provider 5xx error.
- Integration test for interrupted stream behavior.

## Task 7: Chat UI

**Objective**

Build the in-memory chat experience with starter prompts, user/assistant messages, streaming updates, and session-only history.

**Files Affected**

- `components/chat/chat-panel.tsx`
- `components/chat/message-list.tsx`
- `components/chat/chat-input.tsx`
- `app/page.tsx`
- `tests/e2e/upload-dashboard-chat.spec.ts`

**Acceptance Criteria**

- Empty thread shows starter prompts.
- User messages render on submit.
- Assistant response streams into the current thread.
- Refresh clears chat history.
- Error states preserve partial responses when available.
- Chat remains unavailable or CTA-driven when no activities are uploaded.

**Tests Required**

- Component test for empty starter prompts.
- Component test for message submission.
- Component test for streaming assistant deltas.
- Component test for error state.
- Playwright test for submitting a question and seeing streamed output.
