# M4 Polish Task List

Source: `implementation-plan.md`

This milestone prepares the MVP for real use and deployment after the data pipeline, chat, and dashboard flows are functional.

## Task 1: Acceptance Test Suite

**Objective**

Create repeatable acceptance coverage for the MVP questions and guardrails defined in the docs.

**Files Affected**

- `tests/acceptance/fitness-trend.spec.ts`
- `tests/acceptance/monthly-mileage.spec.ts`
- `tests/acceptance/vo2-trend.spec.ts`
- `tests/acceptance/best-aerobic-run.spec.ts`
- `tests/acceptance/fastest-10k-equivalent.spec.ts`
- `tests/acceptance/overtraining-guardrail.spec.ts`
- `tests/acceptance/miles-last-week.spec.ts`
- `supabase/seed.sql`
- `docs/acceptance-tests.md`

**Acceptance Criteria**

- Fitness trend scenario verifies pace trend, HR trend, and confidence language.
- Monthly mileage scenario verifies April-vs-March and highest-mileage-month aggregation.
- VO2 trend scenario handles null values without hallucination.
- Best aerobic run scenario matches the shared efficiency calculation.
- Fastest 10K equivalent scenario returns date, pace, and distance with approximation language when needed.
- Overtraining guardrail refuses diagnosis, coaching, and invented recovery details.
- Miles-last-week scenario converts km to miles and states the date range.

**Tests Required**

- All seven acceptance specs.
- Seed-data setup/teardown for deterministic results.
- Cross-check tests that chat answers match shared calculation outputs.

## Task 2: Upload UX Polish

**Objective**

Make the upload experience clear and resilient for real Garmin CSV imports.

**Files Affected**

- `components/upload/upload-panel.tsx`
- `components/upload/file-dropzone.tsx`
- `app/api/upload/route.ts`
- `tests/integration/upload-route.test.ts`
- `tests/e2e/upload-dashboard-chat.spec.ts`

**Acceptance Criteria**

- Success copy matches the PRD pattern: "`N` runs added, `M` already existed."
- Unrecognized CSV errors tell the user to use Garmin Connect Activities CSV.
- Network and Supabase failures show retryable error copy.
- Re-upload duplicate behavior is understandable.
- Upload progress does not leave the UI stuck after failure.

**Tests Required**

- Component tests for success and error copy.
- Integration tests for route error cases.
- Playwright test for upload retry after failure.
- Playwright test for duplicate re-upload summary.

## Task 3: Chat UX Polish

**Objective**

Make the chat interface feel complete, bounded, and honest about data limitations.

**Files Affected**

- `components/chat/chat-panel.tsx`
- `components/chat/message-list.tsx`
- `components/chat/chat-input.tsx`
- `lib/llm/prompts.ts`
- `app/api/chat/route.ts`
- `tests/integration/chat-route.test.ts`
- `tests/e2e/upload-dashboard-chat.spec.ts`

**Acceptance Criteria**

- Starter prompts are visible when the thread is empty.
- Streaming response state is clear without blocking the UI.
- Partial response plus interruption message appears when a stream fails.
- Chat blocks or redirects to upload CTA when no activities exist.
- Prompt constraints prevent coaching recommendations and training plans.

**Tests Required**

- Component tests for starter prompts and streaming state.
- Integration tests for interrupted stream behavior.
- Integration tests for no-activity behavior.
- Playwright test for a full chat question and streamed response.

## Task 4: Dashboard UX Polish

**Objective**

Ensure dashboard panels handle empty, sparse, and error states without breaking the page.

**Files Affected**

- `components/dashboard/dashboard.tsx`
- `components/dashboard/pace-heart-rate-chart.tsx`
- `components/dashboard/efficiency-trend-chart.tsx`
- `components/dashboard/vo2-trend-chart.tsx`
- `components/dashboard/weekly-mileage-chart.tsx`
- `components/dashboard/recent-runs-table.tsx`
- `tests/e2e/upload-dashboard-chat.spec.ts`

**Acceptance Criteria**

- Each panel has a clear empty or insufficient-data state.
- A failing chart calculation does not blank the full dashboard.
- Recent runs table remains readable with null metrics.
- Dashboard is usable on desktop and mobile viewports.
- No overlapping text or broken layout appears in normal viewport sizes.

**Tests Required**

- Component tests for panel-level empty states.
- Playwright desktop dashboard smoke test.
- Playwright mobile dashboard smoke test.
- Playwright console-error check.
